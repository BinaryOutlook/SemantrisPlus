# Semantris Plus Frontend Enhancement Brief

This brief is for a future AI or contractor who will modernize the current game UI.

Baseline reviewed:

- Current screenshot: `arc/0.12.png`
- Current template: `templates/arcade.html`
- Current styles: `static/css/app.css`
- Current client logic: `static/js/game.js`
- Guidance source: OpenAI, "Designing delightful frontends with GPT-5.4" (March 20, 2026)

The current UI is not bad. It is coherent and readable. The problem is that it frames the game like a retro military console. It feels like a neon command terminal instead of a sharp 2020s puzzle game. The goal is not to make it more decorative. The goal is to make it calmer, flatter, more contemporary, and more obviously game-first.

## 1. Outcome We Want

Transform Semantris Plus from:

- retro-futurist HUD
- scanline-heavy sci-fi dashboard
- telemetry theater
- boxed control-room composition

Into:

- premium 2020s flat puzzle UI
- calm, typography-led, product-grade game surface
- one dominant play area with quiet supporting rails
- minimal but playful motion
- warm, human copy instead of militarized jargon

The finished game should feel like a blend of:

- the clarity of the original Semantris
- the restraint of Linear
- the calm spatial hierarchy of Notion Calendar
- the puzzle-first confidence of NYT Games / Wordle
- the input focus of Superhuman
- the playful feedback discipline of Duolingo

Do not make it look like a SaaS dashboard, a cyberpunk game, or a fake Apple clone. It should feel like a modern word arcade product with its own identity.

## 2. High-Level Diagnosis Of `arc/0.12.png`

What works:

- The center stage is large enough to matter.
- The game state is understandable.
- The target word is visible.
- Score, time, and clue entry are present and usable.
- The overall build already feels more intentional than a rough prototype.

What makes it feel like a "cold war relic":

- The scanlines, vector gates, glows, chamfered corners, and cyan/purple HUD palette signal retro command console.
- The Orbitron-style display typography makes the product feel like a spaceship panel, not a current puzzle game.
- Labels like "Target lock", "AI link", "Provider telemetry", "Iteration command", and "Semantic arcade stack" sound cold, technical, and performatively futuristic.
- Almost every important region is boxed into a panel, so the eye reads chrome before gameplay.
- The top hero ribbon is too tall and too ceremonial for a live game surface.
- The clue panel contains too much explanatory text relative to how simple the action really is.
- Provider details are visually overpromoted for normal players.

Important conclusion:

This is not mainly a color problem. It is a composition, typography, copy, and chrome problem.

## 3. New Creative Direction

### Visual thesis

Semantris Plus should feel like a contemporary editorial puzzle product: flat planes, soft-neutral surfaces, crisp typography, one confident accent color, and a single darkened playfield that makes the words feel tactile and immediate.

### Interaction thesis

Every action should feel fast and legible: type a clue, submit once, watch the stack settle with precise motion, and get a small but satisfying sense of consequence when the target clears.

### Copy thesis

Use human, calm, product-like language. Replace faux tactical jargon with short puzzle language. The UI should sound confident, not theatrical.

### Mood keywords

- precise
- playful
- contemporary
- quiet
- tactile
- intelligent
- game-first

### Anti-keywords

- cyberpunk
- HUD
- command console
- military
- scanline
- telemetry wall
- arcade cabinet nostalgia
- faux sci-fi

## 4. Core Principles To Carry Forward

These are the overarching parameters. Treat them as non-negotiable.

### 4.1 One dominant workspace

The tower is the hero. The first screen should read as one primary play surface with supporting context, not as a collection of equal-sized boxes.

### 4.2 Flat over armored

Use flat color planes, subtle tone shifts, restrained borders, and very light shadow. Remove beveled, clipped, notched, or armored shapes.

### 4.3 Typography over chrome

Hierarchy should come from scale, spacing, alignment, and weight before borders, glows, gradients, or decorative devices.

### 4.4 One accent color, used with discipline

Keep one core accent for the target, primary action, and key progress moments. Danger and error states can use a separate semantic color, but the UI should never feel rainbow-coded.

### 4.5 Utility copy over lore copy

This is a live game surface, not a marketing landing page. Copy should orient, confirm, or invite action. If a sentence sounds like world-building or brand fluff, cut it.

### 4.6 Motion only where it clarifies state

Keep the good board animation direction. Remove ambient "look busy" motion. Use motion to show reorder, clearance, progress, and feedback.

### 4.7 Preserve the original Semantris clarity

The user should understand in seconds:

- what the target is
- where the clue goes
- which area clears words
- whether the last action helped

## 5. Reference Stack And Exact Borrowing Rules

Use these as directional references. Do not clone them literally.

| Reference | What to borrow | Where to apply in Semantris Plus |
| --- | --- | --- |
| [Original Semantris](https://experiments.withgoogle.com/semantris) | Mechanic clarity, direct puzzle framing, semantic movement as the star | Preserve the simple "target + clue + reordering" readability |
| [Linear](https://linear.app/) | Calm hierarchy, thin chrome, dense-but-readable utility UI, disciplined accent use | Rework topbar, side rails, badges, and spacing rhythm |
| [Notion Calendar](https://www.notion.com/product/calendar) | Spacious modern layout, subdued surfaces, clear primary workspace | Make the shell feel like a premium product instead of a themed toy |
| [NYT Games / Wordle](https://www.nytimes.com/games/wordle/index.html) | Puzzle-first confidence, low chrome, strong state contrast | Simplify the board area, labels, and instructions |
| [Superhuman](https://superhuman.com/) | Command-driven input focus and strong single-action UX | Redesign the clue composer into a premium command bar |
| [Duolingo](https://www.duolingo.com/) | Friendly progress signals, compact success feedback, motivational micro-interactions | Improve score feedback, progress bar, and target-clear response |
| [Apple Human Interface Guidelines](https://developer.apple.com/design/human-interface-guidelines/) | Legibility, safe spacing, touch target discipline, reduced visual noise | Mobile behavior, control sizing, hierarchy, and accessibility |

### Borrowing notes

- Borrow Linear's restraint, not its exact grayscale palette.
- Borrow Wordle's confidence in simplicity, not its literal tile system.
- Borrow Duolingo's celebratory timing, not its mascot energy.
- Borrow Superhuman's composer focus, not its enterprise tone.
- Borrow Apple-level cleanliness, not Apple-brand mimicry.

## 6. AI Execution Rules Derived From The OpenAI Frontend Guidance

The future AI implementing this should follow these rules:

- Start from the current screenshot and current files, not from a blank-canvas hallucination.
- Commit to one strong visual thesis, not three half-baked directions.
- Be specific about composition, spacing, color, and motion.
- Use high-quality references and say exactly what is being borrowed from each.
- Remove weak decorative flourishes before adding new ones.
- Make the first viewport feel intentional immediately.
- Prefer hierarchy, contrast, and alignment before gradients and effects.
- Keep the system small: two typefaces max, one accent color, one dominant stage.
- Treat the game screen as an app surface, not a landing page.
- Polish last: first fix structure, then visual language, then motion.

## 7. Visual System Specification

### 7.1 Theme direction

Default to a light-neutral shell with a darker playfield.

Reason:

- It instantly breaks away from the current retro console tone.
- It makes the game look more 2020s product-grade.
- It gives the board a natural focal contrast without depending on neon.

Do not keep the current full-screen dark sci-fi treatment as the primary direction.

### 7.2 Recommended token set

Use a semantic token system like this:

```css
:root {
  --bg: #f5f7fb;
  --bg-subtle: #eef2f7;
  --surface: #ffffff;
  --surface-muted: #f8fafc;
  --playfield: #111827;
  --playfield-elevated: #172033;
  --line: rgba(16, 24, 40, 0.08);
  --line-strong: rgba(16, 24, 40, 0.16);
  --text: #101828;
  --text-muted: #667085;
  --text-inverse: #f8fafc;
  --accent: #14b8a6;
  --accent-strong: #0f766e;
  --danger: #f97316;
  --danger-soft: rgba(249, 115, 22, 0.14);
  --success: #22c55e;
  --focus: #2563eb;
  --radius-sm: 12px;
  --radius-md: 16px;
  --radius-lg: 22px;
  --shadow-sm: 0 1px 2px rgba(16, 24, 40, 0.04);
  --shadow-md: 0 10px 30px rgba(16, 24, 40, 0.08);
}
```

Do not reintroduce:

- neon cyan glow as the main look
- purple secondary glow clouds
- clip-path panel corners
- scanline overlays
- heavy blur everywhere

### 7.3 Typography

Replace the retro display-first type system.

Recommended pairing:

- Primary UI and display: `Manrope` or `Space Grotesk`
- Monospace only for latency, timer, tiny diagnostics: `IBM Plex Mono`

Rules:

- Drop `Orbitron` as the main display typeface.
- Avoid all-caps unless the label is truly micro-utility.
- Reduce letter-spacing globally.
- Use tabular figures for score and timer.
- Let the title be strong because of weight and size, not because it looks like a spaceship.

### 7.4 Shape language

Use:

- rectangles with 12 to 22 px radius
- clean separators
- flat pills where helpful
- one soft shadow layer at most

Do not use:

- notched corners
- diagonal clipping
- faux industrial panel cuts
- ornamental linework

## 8. Content And Copy Rewrite

Replace cold or theatrical labels with clear puzzle language.

Suggested replacements:

| Current | Replace with |
| --- | --- |
| Mode Select | Back |
| New Run | New game |
| Semantic arcade stack | Remove entirely |
| Live semantic cascade | Remove entirely |
| Target lock | Target |
| Vocabulary progress | Progress |
| Turns | Turns |
| Remaining | Remaining |
| AI link | Ranking |
| Provider telemetry | Model status |
| Iteration command | Clue |
| One clue. One reorder. | Type one clue to pull the target lower. |
| Transmit a clue | Type a clue |
| Strike tray | Clear zone |
| Drag the target here to clear | Land the target here to clear it |

Copy rules:

- Prefer sentence case.
- Keep labels to one or two words.
- Limit explanatory copy to one short sentence near the composer.
- Do not narrate the technology unless the user opens a debug view.

## 9. Layout Blueprint

### 9.1 Global composition

Desktop layout should read like this:

- top utility bar
- left support rail
- dominant center playfield
- right action rail

Recommended desktop grid:

```text
220-260 px | flexible center | 280-320 px
```

The center stage must visually dominate.

### 9.2 Top bar

Current issue:

The top area plus hero ribbon consumes too much height and behaves like a dramatic title card.

Change:

- Compress the top area into a compact utility bar.
- Keep brand, current mode, and `New game`.
- Move score and timer into a slim HUD row or stage header, not a giant hero banner.
- Make the header feel like product chrome, not spectacle.

Target feel:

- closer to Linear or Notion utility bars
- less like a game splash screen

### 9.3 Left rail

Keep only the information that supports decision-making:

- current target
- progress
- score or turns if not already in the header

De-emphasize or collapse:

- provider details
- long metadata lists
- low-value labels

The left rail should feel like a quiet support column, not a second dashboard.

### 9.4 Center stage

This is the heart of the redesign.

Requirements:

- The tower should start higher on the screen.
- The board should occupy more of the first viewport.
- The safe zone and clear zone should be legible with flat tonal contrast, not glow effects.
- The target should be recognizable instantly without dominating every other word visually.

Recommended treatment:

- Deep charcoal or midnight playfield
- subtle internal gradient only if necessary
- thin ruled horizontal guides or an ultra-soft grid
- a flat tinted clear zone at the bottom 18 percent to 22 percent
- no luminous sci-fi aura

The stage should feel like a premium puzzle board, not a reactor chamber.

### 9.5 Right rail

This should be the action dock.

Keep:

- clue input
- submit button
- short helper line
- compact status or last-result message

Reduce:

- verbose instructions
- duplicate explanations
- oversized provider box

The input should feel like a command bar, not a form inside a themed panel.

## 10. Selector-Level Implementation Guidance

These instructions are intentionally specific to the current codebase.

### `templates/arcade.html`

- Remove `page-scanlines`, `page-glow`, and `vector-gate` decorative layers from the DOM.
- Collapse the current `hero-ribbon` into a smaller HUD structure or merge its useful metrics into `game-topbar` and `stage-topline`.
- Keep the target visible near the board, not buried in a decorative left card.
- Move provider and latency to a tiny status row, tooltip, or debug-only element.
- Reduce explanatory text inside `.command-card` to one sentence.
- Consider making the clue composer the first thing in the right rail.

### `static/css/app.css`

- Delete or neutralize `--panel-clip` and `--button-clip`.
- Remove clip-path from `.mode-card`, `.preview-stage`, `.hero-ribbon`, `.stage-panel`, `.panel-card`, `.command-button`, `.ghost-link`, and `.clue-input`.
- Remove or fully redesign `.page-scanlines`, `.page-glow`, `.vector-gate`, and their animation rules.
- Rebuild `.hero-ribbon` to be compact, quiet, and structural rather than dramatic.
- Redefine `.panel-card` to rely on tone and spacing instead of blur, gradient, and shadow weight.
- Flatten `.command-button` to a solid fill with a subtle hover lift or inset state, not a neon gradient slab.
- Redesign `.tower-stage` and `.danger-zone` as flat planes with restrained contrast.
- Make `.word-chip` simpler, flatter, and more tactile.
- Use a single consistent radius system across cards, chips, buttons, and panels.

### `static/js/game.js`

Frontend behavior is already structurally solid. Do not rewrite the game loop unnecessarily.

Adjust only what supports the visual system:

- update button and busy-state copy if wording changes
- support any new compact status treatment
- keep FLIP-style reordering and spawn behavior
- refine timing values to match the flatter visual language
- ensure reduced-motion support remains first-class

## 11. Word Stack Design

The words are the product. Treat them with more care.

### Word chip rules

- Each word chip should feel like a clean physical label, not a sci-fi cartridge.
- Use soft corners and a clear fill.
- Neutral words should sit quietly on the stage.
- The target word should use the accent color as a confident fill, not a glow.
- Danger-zone words can receive a subtle tonal shift or thin underline, but do not turn the entire board into warning mode.

### Recommended states

- default word: dark surface, low chrome, high legibility
- target word: solid accent fill, strong text contrast
- danger-zone word: soft warning tint or edge treatment
- just-cleared word: short collapse and dissolve
- newly spawned word: slight upward origin and settle

### Board readability rules

- Ensure chip height is comfortably readable at dense tower sizes.
- Maintain consistent vertical rhythm.
- Keep enough left padding that words do not feel cramped.
- Do not overstyle long words; truncation should be avoided if possible.

## 12. Motion Brief

Motion should be subtle but unmistakable.

Keep these motions:

- reorder travel
- removal burst or dissolve
- collapse into reclaimed space
- spawn from top
- button and input state transitions

Remove or reduce:

- ambient scanline movement
- decorative background drift
- ornamental line pulsing

Recommended timing:

- reorder: 160 to 220 ms
- collapse: 140 to 180 ms
- clear-success accent burst: 160 to 240 ms
- spawn stagger: 30 to 50 ms between chips
- button hover: 120 to 160 ms

Recommended easing:

- `cubic-bezier(0.22, 1, 0.36, 1)` for reorders and entrances
- short ease-out for hover and state changes

The emotional target is "precise and satisfying", not "loud and explosive".

## 13. Responsive Behavior

Mobile should not be a squeezed desktop.

### Tablet and small laptop

- Keep three zones if space allows, but tighten the rails.
- Let the center stage remain dominant.
- Drop low-priority metadata first.

### Mobile

- Move to a single-column layout.
- Order should be: top utility row, target and core stats, stage, status, sticky clue composer.
- Make the composer sticky to the bottom safe area.
- Hide provider latency behind an expandable details row or omit it entirely.
- Preserve touch targets at 44 px minimum.

The mobile version should feel like a polished game app, not a collapsed admin panel.

## 14. Accessibility And Legibility Requirements

Non-negotiable:

- strong color contrast for text and interactive elements
- 44 px minimum interactive height for mobile controls
- visible focus states for keyboard navigation
- reduced-motion path for all board transitions
- no state communicated by color alone
- timer, score, target, and submit action readable in under two seconds

The flat design direction must not become faint, washed out, or low-contrast.

## 15. What Must Be Removed

The future AI should explicitly remove or redesign these elements:

- full-screen scanlines
- vector-gate side ornaments
- broad glow clouds
- clipped sci-fi panel corners
- retro display font as the primary identity
- overly technical labels
- giant hero ribbon that steals vertical space from gameplay
- overemphasized provider telemetry
- dense blocks of instructional copy

If any of these remain mostly intact, the redesign has failed.

## 16. Definition Of Done

The redesign is successful only if all of the following are true:

- The first thing the eye sees is the board, not the chrome.
- The game feels contemporary even with all shadows removed.
- The UI reads as flat and premium, not retro and themed.
- The target, score, timer, and clue input are instantly discoverable.
- The action rail feels focused and elegant.
- The screen no longer looks like a sci-fi dashboard.
- Another designer could identify the influence of Linear, Wordle, and modern product UI without mistaking the game for a clone of any one of them.

## 17. Recommended Implementation Order

1. Remove retro decorative layers and clip-path language.
2. Rebuild layout hierarchy so the stage dominates.
3. Replace typography, color tokens, and shape system.
4. Rewrite labels and helper copy.
5. Flatten and simplify cards, badges, buttons, and chips.
6. Polish board states and motion timing.
7. Refine responsive behavior and accessibility.

Do not start with micro-polish before the structure changes.

## 19. Final Quality Bar

Ask this before shipping:

- Would this still feel modern if all gradients were reduced by 80 percent?
- Would the board still dominate if every border vanished?
- Is the UI calm enough that the words feel important?
- Is the copy short enough to scan instantly?
- Does the screen feel like a game people want to touch, not a dashboard people have to interpret?

If the answer to any of these is "no", keep refining.
