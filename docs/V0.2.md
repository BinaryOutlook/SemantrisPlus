# Semantris Plus V0.2

## Summary

`V0.2` is the frontend migration release for Semantris Plus.

This version does not replace Flask or move the project into a separate SPA framework. Instead, it changes the browser-side implementation from a single untyped JavaScript file into a small TypeScript source tree compiled into a browser bundle.

The goal of this version was straightforward:

- keep the current UI and gameplay flow recognizable
- improve maintainability of the interactive client
- make the frontend safer to change as animation, HUD state, and API usage continue to grow
- establish a more professional build and test path for browser code

In other words, `V0.2` is a frontend engineering upgrade more than a product redesign.

## Why This Update Happened

After the `V0.1` structural cleanup and the `V0.1.2` UI refresh, the frontend had reached the point where plain JavaScript was becoming a weaker fit for the project.

The browser client was responsible for:

- DOM binding
- HUD updates
- API requests
- animation sequencing
- local busy-state handling
- turn orchestration between ranked, removed, and spawned words

That is still a compact codebase, but it is no longer “just a few lines of glue.”

Two technical pressures made the migration worthwhile:

### 1. The frontend now depends on a real API contract

The game view consumes structured state and turn payloads from Flask. Once the client depends on many fields such as score, danger-zone state, provider metadata, and run progression, explicit typing becomes much more valuable.

TypeScript helps by making those payload shapes visible and checked during development rather than being silently trusted at runtime.

### 2. The browser logic now has enough state and sequencing to justify stronger tooling

The game client is not just rendering static markup. It coordinates:

- timers
- disabled states
- board reorders
- explosion effects
- spawn animations
- post-turn status messages

That kind of behavior is manageable in plain JavaScript, but it becomes safer and easier to extend when DOM references, function inputs, and API payloads are typed.

## What Actually Changed

### 1. The interactive client moved into TypeScript source files

The old browser-side path was:

- `static/js/game.js`

The new browser-side source path is:

- `frontend/src/game.ts`
- `frontend/src/controller.ts`
- `frontend/src/api.ts`
- `frontend/src/dom.ts`
- `frontend/src/hud.ts`
- `frontend/src/board.ts`
- `frontend/src/animations.ts`
- `frontend/src/types.ts`
- `frontend/src/state.ts`
- `frontend/src/utils.ts`

This change was made so the frontend could be split by responsibility instead of continuing to grow as one long file.

### 2. The served browser asset became a compiled bundle

The Flask template still serves a browser asset from `static/`, but the asset is now generated:

- source: `frontend/src/*`
- output: `static/js/game.bundle.js`

This means the source of truth is now the TypeScript code, not the generated browser file.

### 3. A frontend build step was introduced

This version adds a small Node-based toolchain:

- `package.json`
- `tsconfig.json`
- `scripts/build-frontend.mjs`

The project now uses:

- `npm install` for frontend dependencies
- `npm run build` to compile the browser client
- `npm run check:frontend` for TypeScript type-checking
- `npm run test:frontend` for frontend tests

This is a meaningful change in project workflow. The repo is no longer purely “run Python and open the browser.” It is now a Flask application with a typed frontend build layer.

### 4. Frontend tests were added

This version introduces frontend-oriented tests through Vitest.

These tests currently cover:

- utility behavior
- DOM reference resolution

This matters because the migration is not only about language preference. It is about creating a safer development path for future frontend work.

### 5. The UI was intentionally kept close to the current version

`V0.2` was not meant to be a visual reset.

The migration deliberately preserves:

- the existing page shell in `templates/arcade.html`
- the current visual system in `static/css/app.css`
- the current Flask API surface
- the same overall interaction flow for loading state, submitting clues, and resolving turns

That was an important design constraint. The project needed a safer frontend foundation without paying the cost of a simultaneous redesign.

## Architecture Impact

### Backend responsibilities stayed mostly the same

Flask still owns:

- template rendering
- session state
- game state serialization
- API routes
- ranking-provider integration

That means the migration is not “Python out, TypeScript in.”

The accurate description is:

- Python remains the backend and delivery layer
- TypeScript now owns the interactive browser implementation

### Frontend responsibilities became more explicit

The client is now easier to understand in slices:

- `dom.ts`: required element lookups and DOM guards
- `types.ts`: shared frontend contract types
- `api.ts`: JSON request and response handling
- `hud.ts`: score, timer, provider, and status rendering
- `board.ts`: word-chip rendering and classification
- `animations.ts`: transition and explosion behavior
- `controller.ts`: page orchestration and event handling

This separation is the main maintainability win of `V0.2`.

## Workflow Changes

Running the project now involves both Python and npm.

Typical local flow:

```bash
python3 -m pip install -r requirements.txt
npm install
npm run build
python3 app.py
```

Recommended validation flow:

```bash
npm run check:frontend
npm run test:frontend
python3 -m unittest discover -s tests
```

If the TypeScript source changes, the bundle should be rebuilt before running the app or preparing a change for review.

## Why The Structure Looks This Way

The project still aims to stay small and readable.

That is why `V0.2` does not introduce:

- React
- Vue
- a client router
- a state management framework
- a large asset pipeline

The chosen approach is intentionally conservative:

- keep Flask
- keep Jinja shells
- keep the CSS system
- add typed frontend modules
- add only the build tooling needed to support them

This gives the repo a stronger engineering base without overshooting its current size.

## Known Remaining Gaps

Even after `V0.2`, there is still work left:

- richer frontend test coverage beyond utilities and DOM setup
- smoother developer ergonomics such as a dedicated watch mode if desired
- more explicit deployment instructions for environments that must build the frontend bundle
- continued motion and end-of-run polish

## Recommended Next Step

The best next step after `V0.2` is not another structural migration.

The next best step is to build on the new frontend foundation:

- expand frontend test coverage around gameplay flows
- keep improving motion polish
- improve run completion UX
- continue strengthening documentation for contributors

`V0.2` should reduce the risk of those changes by making the browser-side code easier to reason about and safer to extend.
