# Semantris Plus V0.1

## Summary

`V0.1` is the first structural cleanup release for Semantris Plus.

This update is not just a visual refresh. It is the point where the project stops being a single-file prototype and starts becoming a maintainable small game repository. The goal of this version was to improve both the player-facing experience and the engineering quality underneath it.

## Why This Update Happened

The original prototype proved the core idea, but it had several problems:

- gameplay rules, server behavior, and provider integration were tightly coupled
- the frontend lived almost entirely inside one template
- session state was too thin for a good arcade run
- word reuse and provider failures were not handled robustly
- documentation was not strong enough for future contractor work

This update was designed to fix that foundation before continuing with more polish.

## Player-Facing Changes

Players should now notice:

- a more intentional tower-focused interface
- a clearer HUD with score, time, turns, provider, latency, and vocabulary progress
- a visible destruction zone at the bottom of the tower
- smoother sequencing for reorder, removal, and drop-in events
- more stable session progression across a run

The tower is also explicitly oriented so the bottom-most word represents the strongest semantic match for the latest clue.

## Backend and Systems Changes

### 1. Gameplay logic was separated from Flask routes

Gameplay rules now live in `game_logic.py`.

This was done so the board mutation logic can be tested independently of the web server. It also makes future gameplay changes less risky because they no longer require editing route handlers and UI code at the same time.

### 2. LLM integration was isolated

Provider interaction now lives in `llm_client.py`.

This was done so prompt changes, output parsing, validation rules, and future provider migrations can happen without rewriting the whole app. It also allowed the addition of a fallback ranking path.

### 3. Session state became a real contract

The app now tracks more than score and board contents. It also tracks run metadata such as:

- turns
- elapsed time
- last clue
- last latency
- provider used
- vocabulary progress
- whether fallback ranking was used

This makes the frontend easier to drive and gives the project a better base for future replay, analytics, or leaderboard work.

### 4. No-repeat word handling was improved

The game now avoids reusing words within a run until the unseen vocabulary pool is exhausted.

Structurally, this matters because it makes the run feel more legitimate and less prototype-like. It also gives the game a clearer sense of progression through a vocabulary set.

### 5. The app now has a clearer API surface

Instead of relying only on a rendered page plus one implicit flow, the project now exposes clearer endpoints for:

- loading game state
- starting a new run
- resolving a turn

This was chosen because the frontend needs stable, structured data to animate properly and because a better API surface makes future extensions easier.

## Frontend Structure Changes

### 1. The template became a shell

`templates/arcade.html` now acts as the page shell instead of containing the entire application implementation.

### 2. CSS and JavaScript were moved into static assets

- `static/css/app.css`
- `static/js/game.js`

This was done for maintainability and clarity. Once the project started needing a more serious UI and animation system, keeping everything inside one template would have made iteration unnecessarily brittle.

### 3. The UI was rebuilt around explicit play areas

The layout is now divided into:

- information/HUD panel
- tower stage
- clue input area
- progress and metadata panels

This was chosen because industrial-quality game repos typically separate the concerns of presentation, interaction, and state display instead of blending everything together in one flat layout.

## Documentation Changes

This version also introduces better documentation structure:

- `README.md` is now the primary project-facing document
- `brief.md` is the contractor-facing product and engineering brief
- `docs/V0.1.md` records this update and the rationale behind it

This separation was intentional. Different documents serve different audiences, and combining all of them into one README would make the repo harder to maintain.

## Testing Changes

Automated tests were added for:

- board size behavior
- session initialization
- hit resolution
- miss resolution
- API response structure

The project is still small, but adding tests at this stage was important so future UI and provider work does not silently break the core loop.

## Why The Structure Looks This Way

The current structure is deliberately simple, not enterprise-heavy.

The project now separates:

- web delivery
- gameplay domain logic
- provider integration
- presentation assets
- tests
- long-form documentation

That separation gives the repo a better long-term shape without overcomplicating a small game.

In other words, the goal was not to make the project “big.” The goal was to make it understandable, stable, and extensible.

## Known Remaining Gaps

Even after `V0.1`, there is still important work left:

- migrate away from the deprecated Gemini SDK
- continue improving motion quality
- improve fallback semantic quality
- add richer completion states
- add persistence or competitive features if desired

## Recommended Next Step

The best next version after `V0.1` is likely a polish-focused release:

- better end-of-run UX
- stronger audiovisual feedback
- more configurable run modes
- provider modernization

That would build on the structural work in this version instead of having to redo foundation work later.
