## Purpose

This note records the tech-stack and tooling changes that were actually implemented as part of the `v0.4` pass.

The matching proposal document at `docs/proposals/techstack_ideas.md` should now be treated as the list of remaining future-facing ideas rather than a list of already-delivered work.

## What v0.4 Implemented

| Area | Implemented change | Outcome |
| --- | --- | --- |
| Runtime configuration | Added centralized typed settings through `settings.py` using `pydantic-settings` | Runtime config is now validated and easier to reason about than scattered `os.getenv` reads |
| LLM test mode | Added a deterministic fake ranker | Browser and integration tests can now run without live remote model access |
| Ranking resilience | Added a stronger local semantic fallback ahead of the older heuristic fallback | Remote-provider failures now degrade more gracefully |
| Semantic caching | Added a request-level cache abstraction with an in-memory backend | Repeat ranking and scoring calls can now reuse validated results |
| Persistence | Added SQLAlchemy-backed local run persistence with SQLite by default | Completed runs can now be saved and queried for local best-score behavior |
| API payloads | Extended state payloads with additive persistence metadata | The frontend can display local-best information without changing the gameplay contract drastically |
| Frontend UX | Added local-best messaging to game-over flows | Persistence now creates visible player-facing value |
| Frontend quality tooling | Added Biome and a formal `npm run lint` command | Frontend source now has a formal lint/check path |
| Browser E2E coverage | Added Playwright plus deterministic browser-flow tests | The repo now has end-to-end coverage for landing, theme persistence, and all three modes |
| Repo documentation | Updated README and env templates for the new runtime and commands | The new workflows are discoverable from repo entry docs |

## What v0.4 Explicitly Did Not Do

These items were discussed in the proposal but were not implemented in this pass and therefore remain future ideas:

- Vite migration
- Redis-backed cache or session infrastructure
- embedding-backed retrieval or vector infrastructure
- cross-encoder reranking
- dedicated self-hosted model integrations such as Ollama or vLLM setup
- Sentry integration
- Langfuse or Helicone integration
- GitHub Actions or Docker rollout work
- framework replacements such as FastAPI, Django, React, Vue, or Svelte

## Practical Summary

`v0.4` should be understood as a strengthening pass, not a replatform:

- keep the existing Flask + Jinja + TypeScript architecture
- make ranking more resilient
- make configuration cleaner
- add local persistence
- add formal browser and frontend tooling

That work is now historical. The proposal doc should only carry the unresolved future-facing ideas from this point onward.
