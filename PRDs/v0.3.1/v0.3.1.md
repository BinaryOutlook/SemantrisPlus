## 1. Iteration Summary

`v0.3.1` is a focused frontend iteration for adding a new opt-in visual theme called `Cupertino`.

This is not a new gameplay mode. It is a presentation pass for the web experience that should make the landing page and all playable pages feel more precise, more product-like, and more cohesive while keeping Semantris Plus unmistakably its own game.

The source materials for this packet are the Apple-inspired design references stored in `PRDs/references/v0.3.1-cupertino-source-packet/`. Those files are inspiration inputs, not a literal implementation spec and not a requirement to recreate Apple's site exactly.

## 2. Current Baseline

The repository already has a meaningful frontend baseline:

- a shared frontend bundle with explicit theme handling
- persisted manual theme selection through `semantris-theme`
- light and dark theme support across the landing page and all three game pages
- one shared stylesheet with tokenized colors, spacing, and component styling
- a landing page plus three distinct gameplay shells
- a typed theme controller with test coverage

What the product does not yet have is a third visual identity that feels more controlled and editorial. The current frontend is coherent, but it is still anchored to the existing earthy light/dark design language.

## 3. v0.3.1 Thesis

`v0.3.1` should introduce `Cupertino` as a first-class theme option that translates the Apple-inspired source packet into a Semantris-specific visual mode.

The goal is not imitation for its own sake. The goal is to borrow the useful qualities of that design language:

- disciplined typography
- restrained color usage
- quiet but premium surfaces
- crisp hierarchy
- product-first framing
- minimal but deliberate interaction styling

The result should make Semantris Plus feel cleaner and sharper without:

- hiding the playfield
- weakening readability
- breaking the existing light/dark themes
- turning the site into an Apple clone

## 4. Goals

### 4.1 Product goals

- add `Cupertino` as a visible third theme option on the landing page and every game page
- give the web UI a calmer, more precise product-style presentation
- keep the playfield as the hero while making surrounding controls feel more refined
- align buttons, typography, spacing, surfaces, and focus states under one Cupertino language
- preserve fast start, easy replay, and immediate run-state readability

### 4.2 Engineering goals

- extend the theme system from a binary light/dark model to a clean multi-theme model
- keep theming token-driven instead of creating page-by-page style forks
- preserve current ownership boundaries between templates, CSS, and frontend scripts
- keep the documentation trail clear so future agents can re-enter from `PRDs/v0.3.1/`

## 5. Non-Goals

`v0.3.1` is not trying to deliver:

- gameplay rule changes in Iteration, Restriction, or Blocks
- a frontend framework rewrite
- a full landing-page content rewrite unrelated to theme work
- a literal copy of apple.com
- Apple logos, Apple product photography, or copied Apple marketing copy
- vendored proprietary SF font files
- removal of the existing light and dark themes

## 6. Primary Users

### 6.1 Players

Players should be able to opt into a more polished product-style presentation without losing gameplay clarity.

### 6.2 Primary maintainer

The maintainer should be able to extend or tune the theme system without introducing a separate visual codepath for every page.

### 6.3 Future agents or contributors

Future helpers should be able to understand:

- that `Cupertino` is a theme, not a new game mode
- which source materials informed the theme
- which frontend surfaces must participate in the theme
- which implementation shortcuts would create regressions or documentation drift

## 7. Experience Direction

### 7.1 Theme identity

In `Cupertino`, Semantris Plus should feel:

- calm
- precise
- premium
- restrained
- product-led
- fast to scan

The visual identity should emphasize typography, spacing, contrast, and composition before decoration.

### 7.2 Color behavior

The Cupertino theme should adapt the source packet's color logic to Semantris:

- use black and light gray as the primary environment colors
- use near-black text on light surfaces and white text on dark surfaces
- reserve blue as the main interactive accent
- avoid introducing extra accent colors unless required for game-state semantics such as success, warning, or danger
- keep semantic status colors functional rather than decorative

On the landing page, stronger alternation between light and dark sections is allowed where it improves rhythm. On gameplay pages, readability and state clarity take priority over theatrical contrast.

### 7.3 Typography direction

The theme should move away from the current expressive pairing toward a Cupertino-style system stack:

- prefer `SF Pro Display`, `SF Pro Text`, `-apple-system`, `BlinkMacSystemFont`, `Helvetica Neue`, `Arial`, `sans-serif`
- use the stack as a preference order only; do not vendor proprietary font files
- tighten headline tracking and line height
- keep body copy short, crisp, and readable
- avoid decorative display type in Cupertino mode

### 7.4 Motion and interaction

Motion should stay subtle and functional:

- focus and hover states should feel crisp, not flashy
- transitions should support state change and perceived polish
- no ornamental animation system is required for this iteration

### 7.5 Copy direction

Copy inside Cupertino surfaces should stay short, direct, and product-facing. It should sound confident and clear, not ornate or ad-like.

## 8. In-Scope Product Surfaces

### 8.1 Theme system

The Cupertino theme applies to the web experience as a shared presentation layer. It must participate in:

- landing page theme selection
- Iteration Mode page theme selection
- Restriction Mode page theme selection
- Blocks Mode page theme selection
- persisted cross-page theme preference

### 8.2 Landing page

The landing page should reflect Cupertino most visibly. In scope:

- top bar
- hero typography
- CTA cluster
- mode cards
- pack selection controls
- preview shell and supporting chips or stats

### 8.3 Shared game shell

Each gameplay page should receive Cupertino styling for:

- header and topbar
- theme control
- score, timer, and status areas
- buttons, links, selects, and clue inputs
- panel cards
- game-over and replay surfaces

### 8.4 Mode-specific playfield surfaces

Each mode keeps its current mechanics, but the surrounding presentation should be themed:

- Iteration Mode tower shell and target presentation
- Restriction Mode rule card and strike/status framing
- Blocks Mode board chrome, score framing, and end-state shell

The actual playfield must remain readable first. Cupertino styling should not sacrifice contrast, target clarity, or input clarity just to preserve visual purity.

### 8.5 Responsive behavior

The theme must work on desktop and mobile widths. The theme selector and primary CTAs must remain easy to use on smaller screens.

## 9. Functional Requirements

### 9.1 Theme selection and persistence

- The frontend must support three explicit theme values: `light`, `dark`, and `cupertino`.
- The shared storage key remains `semantris-theme`.
- A user-selected `cupertino` theme must persist across the landing page and all game pages.
- If no stored preference exists, the app should continue following the system light/dark preference rather than auto-selecting Cupertino.
- The active theme must remain visible in the theme control with a clear selected state.

### 9.2 Cupertino token system

- The stylesheet must define a dedicated token override for `:root[data-theme="cupertino"]`.
- Cupertino tokens should cover at minimum:
  - font families
  - background and surface colors
  - text colors
  - interactive accent and focus colors
  - playfield and panel surfaces
  - radii
  - shadows
- The accent blue should be used primarily for interactive elements and focus treatment.
- Landing and gameplay tokens may differ where necessary, but the overall theme language must remain coherent.

### 9.3 Component styling requirements

In Cupertino mode, the design direction should include:

- compact but readable typography
- tighter headline line heights
- refined button styling with softer radii or pill treatment where appropriate
- restrained card styling with limited borders and sparse shadow use
- a translucent or glass-like top navigation treatment if it remains readable and technically safe
- clear focus states on all interactive elements

### 9.4 Landing page presentation requirements

- The landing page should feel more product-led than dashboard-like in Cupertino mode.
- The hero should explain the game quickly with concise copy and clearer visual hierarchy.
- The primary action area should feel immediate and polished.
- Mode preview cards should feel like one family rather than disconnected widgets.
- The theme should support a stronger cinematic rhythm without burying the pack selector or start actions.

### 9.5 Game page presentation requirements

- Score, elapsed time, turns, and mode identity must stay easy to scan at a glance.
- Clue entry and submission controls must remain visually dominant enough to guide the player's next action.
- Status, warning, success, and failure messaging must preserve contrast and urgency.
- End-of-run surfaces should feel more intentional and product-quality, even if the interaction model stays the same.
- The playfield should remain the visual hero over surrounding chrome.

### 9.6 Accessibility and compatibility requirements

- Keyboard focus must remain obvious in Cupertino mode.
- Interactive contrast must meet a usable accessibility bar on both light and dark surfaces.
- The theme must degrade gracefully on non-Apple devices and browsers.
- The theme must not rely on proprietary fonts being locally available in order to remain usable.
- The theme must not depend entirely on Safari-only behavior.

### 9.7 Documentation continuity

- `PRDs/v0.3.1/v0.3.1.md` should remain the canonical iteration brief.
- The source design packet in `PRDs/references/v0.3.1-cupertino-source-packet/` should be treated as reference material.
- Post-implementation change notes should be appended to the bottom of this version PRD.

## 10. Technical Guardrails

### 10.1 Ownership boundaries

- `frontend/src/theme.ts` owns theme resolution, persistence, and theme control behavior.
- `templates/*.html` own the early theme bootstrap and theme-toggle markup, not theme logic duplication.
- `static/css/app.css` owns theme tokens and theme-specific component styling.
- Gameplay controllers should only change where presentation hooks are genuinely required.

### 10.2 Theme architecture discipline

- Do not create separate Cupertino-only templates unless there is a compelling reason.
- Do not fork the whole stylesheet into one file per theme.
- Prefer shared component styles with token overrides and a small number of well-scoped theme-specific selectors.
- Keep theme logic explicit and typed; do not allow unvalidated string themes to spread through the frontend.

### 10.3 Brand and asset discipline

- Do not use Apple trademarks, Apple logos, or copied Apple product imagery.
- Do not package proprietary SF font assets with the repo.
- Inspiration from the source packet should be translated into Semantris Plus rather than copied literally.

### 10.4 Regression discipline

- Existing `light` and `dark` themes must continue to work after Cupertino is added.
- Theme tests should expand to cover the third option rather than narrowing to Cupertino only.
- The theme pass must not quietly change gameplay contracts or backend behavior.
