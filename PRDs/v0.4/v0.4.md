# Semantris Plus v0.4

Status: Future iteration brief
Version folder: `PRDs/v0.4/`
Related foundation doc: `docs/PRD.md`
Related proposal: `docs/proposals/techstack_ideas.md`
Date: 2026-04-09

## 1. Iteration Summary

`v0.4` is a future-facing engineering and product-readiness iteration built from the high-priority and medium-priority recommendations in `docs/proposals/techstack_ideas.md`.

This is not a framework rewrite. It is a targeted strengthening pass that should make Semantris Plus:

- more reliable when ranking providers fail or vary
- easier to test end to end
- easier to configure safely
- more ready for persistent run history and local leaderboard features
- more maintainable for future AI coding agents

The main idea is simple:

keep the current Flask + Jinja + TypeScript architecture, but make the semantic engine, tooling, and persistence layer much stronger.

## 2. Current Baseline

Before `v0.4`, the repository already has:

- a modular Flask backend
- three playable modes: Iteration, Restriction, and Blocks
- mode-specific gameplay logic modules
- a TypeScript frontend source tree bundled into static browser assets
- Gemini and OpenAI-compatible ranking support behind `llm_client.py`
- a heuristic fallback path when remote ranking fails
- backend unit tests and frontend Vitest coverage
- a documentation structure with stable PRD and versioned iteration packets

What the repo does not yet have is a more mature technical spine for the next phase of product work:

- typed centralized runtime settings
- browser-level end-to-end test coverage
- stronger semantic ranking fallback and caching
- durable run persistence
- a clearer future path for frontend developer tooling

## 3. v0.4 Thesis

`v0.4` should make Semantris Plus feel less like a clever prototype with good structure and more like a small product with dependable technical behavior.

The core bet of this version is:

1. Better semantic quality and fallback quality will improve the actual game more than a major replatform would.
2. Better test coverage and typed configuration will reduce future change risk more than adding more surface-level polish alone.
3. A minimal persistence layer will unlock more product value than a frontend framework migration.

## 4. Goals

### 4.1 Product goals

- improve the trustworthiness of ranking results and fallback behavior
- make repeated local play more rewarding through saved run history or best-score tracking
- preserve the current fast-to-start local browser experience
- keep the three modes visually and mechanically intact while strengthening the infrastructure underneath them

### 4.2 Engineering goals

- add typed centralized configuration
- add semantic-result caching and a stronger local semantic fallback path
- add deterministic browser-level test coverage
- add a persistence foundation using SQLAlchemy with SQLite by default and Postgres compatibility
- optionally modernize frontend build ergonomics with Vite if the migration stays low-risk and preserves the current served-asset model

## 5. Non-Goals

`v0.4` is not trying to deliver:

- a React, Vue, Svelte, or Next.js replatform
- a Django migration
- accounts, cloud sync, or multiplayer
- a generic AI orchestration framework
- a prompt-chaining platform
- a major redesign of the landing page or game shells
- gameplay rule changes to Iteration, Restriction, or Blocks
- removal of Gemini or OpenAI-compatible support

## 6. Primary Users

### 6.1 Players

Players should get more dependable runs, clearer saved progress signals, and fewer cases where model behavior makes the game feel erratic.

### 6.2 Primary maintainer

The maintainer should be able to re-enter the repo and understand configuration, persistence, and test workflows quickly.

### 6.3 Future AI coding agents

Future coding agents should be able to read this packet and understand:

- what must be built
- which new modules should exist
- which files own which responsibilities
- how the system should be tested before a merge is considered complete

## 7. v0.4 Pillars

### 7.1 Semantic engine quality

The semantic engine should get stronger without replacing the current provider abstraction.

Required direction:

- keep Gemini and OpenAI-compatible ranking support
- add request-level caching for repeatable ranking operations
- improve local fallback quality beyond the current simple heuristic-only safety net
- keep provider isolation inside the ranking layer

### 7.2 Confidence and testing

The repo should move from unit-test confidence to workflow confidence.

Required direction:

- add deterministic Playwright coverage
- keep current backend and frontend unit tests
- add linting for the TypeScript frontend
- make browser test runs independent from live remote model access

### 7.3 Typed settings and runtime clarity

Configuration should become explicit and validated at startup.

Required direction:

- move important env parsing into a typed settings module
- keep one clear source of truth for runtime configuration
- fail early on invalid configuration rather than failing deep in the request path

### 7.4 Persistence foundation

The product should gain durable run history and local score persistence.

Required direction:

- introduce SQLAlchemy
- default to SQLite for local development
- remain compatible with a future Postgres deployment
- keep persistence minimal and focused on runs and scores, not accounts

### 7.5 Developer loop improvement

The frontend build path should be evaluated for smoother iteration.

Required direction:

- preserve the current no-framework frontend model
- keep multi-entry browser bundles
- only adopt Vite if it stays low-risk and does not create confusing Flask integration

## 8. In-Scope Workstreams

### 8.1 Workstream A: Typed settings and startup validation

`v0.4` should add a typed settings layer using `pydantic-settings`.

The purpose is to replace scattered runtime parsing with one validated configuration object.

Required configuration areas:

- Flask/session secrets
- selected LLM provider
- Gemini credentials and model name
- OpenAI-compatible credentials, base URL, and model name
- cache backend settings
- persistence settings
- test-only fake-ranker settings
- optional frontend tooling flags if needed

Recommended new module:

- `settings.py`

Recommended design:

- one root settings object with grouped sub-settings where helpful
- default values for local development where safe
- explicit validation for allowed provider names and backend selections
- startup-time creation and reuse of one settings object

### 8.2 Workstream B: Semantic caching and stronger fallback quality

The current ranking layer should remain provider-agnostic, but gain two new abilities:

1. cache repeated semantic operations
2. use a better local fallback path when remote ranking fails

### Required semantic operations to support

The implementation must cover all ranking-like operations already owned by `llm_client.py`, including:

- standard word ranking
- restriction-mode rule judgment and ranking
- blocks primary candidate selection
- blocks candidate scoring

### Required cache behavior

- identical semantic requests should be normalized into stable cache keys
- cache writes should happen only after successful validated results
- cache hits must not change gameplay semantics
- the app must remain usable if the cache backend is unavailable
- the default local development mode may use `none` or in-process memory caching
- Redis support should be available for stronger caching in future deployments

### Required fallback behavior

When remote ranking fails, the system should attempt:

1. a stronger local semantic fallback if enabled
2. the current heuristic fallback if the stronger local path is unavailable

The stronger fallback should be local and deterministic enough for testing.

Recommended approach:

- add an embedding-based fallback ranker using a sentence-transformer-style embedding model
- compute clue-to-word similarity locally
- rank or score candidates locally when the remote path fails

If the embedding model cannot initialize:

- log a warning
- continue safely with the existing heuristic fallback

### Important rule

Do not introduce LangChain, LlamaIndex, or a larger orchestration layer in `v0.4`.

### 8.3 Workstream C: Playwright and deterministic browser test mode

The repo should gain browser-level tests that cover actual user flows without depending on live remote models.

Required design:

- add Playwright configuration and scripts
- add a deterministic fake-ranker mode for browser tests
- keep fake ranking behavior simple, readable, and fully local

Recommended fake-ranker contract:

- enable through an environment flag such as `SEMANTRIS_USE_FAKE_RANKER=1`
- do not require changing the public provider name contract
- return deterministic valid permutations and scores for all three modes

Minimum required browser scenarios:

- landing page loads
- theme controls work and persist
- new run can be started for each mode
- one turn can be submitted successfully in each mode using the fake ranker
- end-of-run or game-over shell behavior can be exercised at least once in a deterministic way

### 8.4 Workstream D: Local persistence and scoreboard foundation

`v0.4` should add durable local persistence for run summaries and high-score queries.

This is not an accounts system.

### Required persistence scope

- save completed run summaries
- save enough metadata to compute local best scores
- support all three modes
- support vocabulary-pack-aware queries

Recommended schema shape:

- `run_records`

Recommended fields:

- `id`
- `mode_id`
- `pack_id`
- `score`
- `turns`
- `elapsed_seconds`
- `game_result`
- `provider_label`
- `used_fallback`
- `created_at`

Optional fields if helpful:

- `words_cleared`
- `strike_count`
- `board_clear_bonus`
- `metadata_json`

### Required product behavior

At minimum, the game should be able to:

- record completed runs
- determine whether the current run is a new local best for the selected mode and pack
- expose best-score or recent-run data to the frontend through explicit API payloads

Recommended minimum UI surfaces:

- game-over summary shows whether the player achieved a new local best
- landing page or mode shell can show a lightweight best-score indicator if the added UI remains clear

### 8.5 Workstream E: Frontend linting and build-tool evaluation

`v0.4` should improve frontend code hygiene and optionally improve frontend build ergonomics.

### Required linting outcome

The repo must gain a formal frontend lint command.

Recommended stack:

- ESLint for TypeScript
- Prettier for formatting

Acceptable alternative:

- Biome, if chosen decisively and applied consistently

### Required Vite evaluation outcome

Vite is not mandatory, but its evaluation is in scope.

If Vite is adopted in `v0.4`, it must:

- preserve the current multi-entry bundle model
- continue producing browser assets served by Flask
- avoid introducing a frontend framework
- preserve or clearly update template asset references
- keep production build output obvious for future contributors

If Vite is not adopted, the PRD still counts as successful if the repo gains:

- frontend linting
- Playwright coverage
- a documented reason for retaining `esbuild`

## 9. Functional Requirements

### 9.1 Settings requirements

- Invalid provider names must fail during startup validation.
- Missing required credentials must produce clear startup errors when the selected provider requires them.
- Settings must be readable from one central settings module.
- Tests must be able to override settings safely without mutating many unrelated globals.

### 9.2 Semantic engine requirements

- Caching must never return structurally invalid results.
- Fallback ranking must preserve current validation discipline.
- Provider-specific logic must remain isolated from gameplay modules.
- Current API payloads must remain backward compatible unless a clearly additive metadata field is introduced.
- Warning and fallback information must remain available to the frontend.

### 9.3 Persistence requirements

- Persistence must be optional and environment-configurable.
- Local development must work with SQLite.
- The schema must be simple enough for future migration to Postgres.
- If persistence is disabled or unavailable, gameplay must still work.

### 9.4 Testing requirements

- Browser tests must run without external LLM calls.
- Python tests must continue covering gameplay rules and route behavior.
- New semantic caching and persistence logic must have unit coverage.
- Frontend linting must be runnable in CI or local validation.

### 9.5 Build requirements

- Frontend bundles must remain explicit and understandable.
- The repo must not end up with two competing sources of truth for frontend build output.
- Any tooling change must preserve the current served-browser-asset workflow.

## 10. Technical Guardrails

### 10.1 Preserve architecture shape

- Keep Flask, Jinja, and the current TypeScript frontend model.
- Do not introduce React, Vue, Svelte, or Next.js as part of `v0.4`.
- Do not split the game into a separate SPA.

### 10.2 Preserve ownership boundaries

- `app.py` should continue owning route wiring, session orchestration, and API serialization.
- `game_logic.py`, `game_logic_restriction.py`, and `game_logic_blocks.py` should continue owning gameplay rules.
- `llm_client.py` or a tightly related ranking module group should own provider integration, caching, parsing, and fallback behavior.
- Persistence code should live in dedicated modules rather than leaking into gameplay files.
- Frontend bundles should continue owning browser orchestration and UI behavior.

### 10.3 Preserve mode boundaries

- Do not flatten Iteration, Restriction, and Blocks into one oversized mode abstraction.
- Shared helpers may be extracted, but only if the result is clearer than duplicated logic.

### 10.4 Preserve documentation discipline

- `docs/PRD.md` remains the stable product foundation.
- `PRDs/v0.4/` is the canonical packet for this iteration.
- Any implementation notes or completion notes should be appended to this file after the work lands.
