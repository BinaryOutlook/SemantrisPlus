# Semantris Plus v0.3

Status: Active iteration brief
Version folder: `PRDs/v0.3/`
Related foundation doc: `docs/PRD.md`
Date: 2026-03-30

## 1. Iteration Summary

`v0.3` is the first versioned iteration packet for Semantris Plus under the new PRD workflow.

The purpose of this version is not to invent a brand new product. The product already exists and is playable. The purpose of `v0.3` is to make the current product feel more deliberate, more coherent, and easier to extend by establishing one clear iteration target that combines:

- product intent
- design direction
- UX priorities
- technical guardrails
- a visual demo artifact future agents can follow

This folder should be treated as the source of truth for the `v0.3` iteration.

## 2. Current Baseline

The repository already has a meaningful foundation:

- three playable modes: Iteration, Restriction, and Blocks
- a Flask backend with explicit JSON state endpoints
- mode-specific gameplay logic modules
- a typed frontend source tree compiled into browser bundles
- deterministic fallback behavior when remote ranking fails
- version notes, tests, and a product foundation document

What is still missing is a stronger iteration spine for future work. The project has enough moving parts that feature work should no longer begin from vague intent alone.

## 3. v0.3 Thesis

`v0.3` should make Semantris Plus feel like a coherent semantic arcade product rather than a collection of promising screens and good engineering decisions.

This version should improve two things at the same time:

1. The player-facing clarity of the product.
2. The builder-facing clarity of the repo and iteration workflow.

## 4. Goals

### 4.1 Product goals

- clarify the product story on the landing page
- unify the visual hierarchy across the three game shells
- make the current run state easier to scan during play
- improve the feel of completion, replay, and status feedback
- keep the playfield as the visual hero

### 4.2 Engineering goals

- establish a versioned iteration workflow under `PRDs/`
- keep mode-specific behavior isolated
- protect the codebase from accidental spaghetti as more polish work is added
- ensure future agents can start from a version folder instead of reconstructing context from many docs

## 5. Non-Goals

`v0.3` is not trying to deliver:

- accounts, profiles, or cloud saves
- multiplayer features
- leaderboards
- a major backend replatform
- a frontend framework rewrite
- a generic plugin system
- every long-term mode idea at once

## 6. Primary Users

### 6.1 Players

Players want a short, replayable, browser-based semantic puzzle they can understand quickly and replay without friction.

### 6.2 Primary maintainer

The repo is still maintained primarily by one builder. Re-entry after time away from the codebase must stay easy.

### 6.3 Future agents or contributors

Future helpers should be able to read the `v0.3` folder and understand:

- what this iteration is trying to change
- what the UX should feel like
- what files are likely to be touched
- what architectural boundaries should be respected

## 7. Experience Direction

### 7.1 Product feeling

Semantris Plus should feel:

- calm
- sharp
- playful
- modern
- tactile
- replay-friendly

### 7.2 Designer-operator baseline

This version should follow these initial design rules:

- use a light-first editorial shell with a darker playfield
- keep the playfield visually dominant
- use one accent color with discipline
- rely on typography, spacing, and surface contrast before decoration
- avoid retro HUD language, dashboard chrome, neon gimmicks, and decorative glass effects
- use motion only where it explains state changes
- make the interface feel intentional on both desktop and mobile

### 7.3 Copy direction

Copy should stay short, human, and game-facing. It should orient the player and reinforce rhythm, not sound like a tech demo.

## 8. In-Scope Product Surfaces

### 8.1 Landing page

The landing page should:

- explain the product quickly
- show the three modes as one family
- keep pack selection visible without clutter
- make starting a run feel immediate
- communicate that the project is a semantic arcade, not a generic AI app

### 8.2 Shared game shell

All game shells should share the same broad interaction logic:

- visible score and elapsed time
- clear mode identity
- obvious clue entry
- visible status feedback
- readable progress and vocabulary state
- a replay or new-run action that does not feel buried

### 8.3 Iteration Mode

Iteration Mode remains the clearest expression of the core loop and should continue to present:

- one current target
- one clue per turn
- a visible clear zone
- obvious cause-and-effect between semantic ranking and board outcome

### 8.4 Restriction Mode

Restriction Mode should feel like a disciplined variant of the same product, not a separate app.

It must keep:

- the active rule always visible
- strike pressure easy to understand
- legal vs. illegal clue outcomes easy to distinguish

### 8.5 Blocks Mode

Blocks Mode should feel mechanically different but visually part of the same family.

It must keep:

- the primary hit easy to identify
- the chain reaction understandable
- gravity and refill visually legible

### 8.6 End-of-run states

Each mode should move toward clearer completion feedback.

At minimum, the design direction for `v0.3` should support:

- stronger completion messaging
- better summary of score, time, and turns
- a more satisfying replay prompt

## 9. Functional Requirements

### 9.1 Iteration workflow

- `v0.3` must exist as a version folder under `PRDs/`.
- The folder must contain a PRD and a demo HTML file.
- Future coding work for this version should start by reading both files together.

### 9.2 Product clarity

- Mode purpose should be understandable from the landing page without reading long paragraphs.
- During play, the primary action and the most important run state must be visible immediately.
- The interface should reduce scanning effort rather than distribute attention equally across too many surfaces.

### 9.3 Reliability

- Provider failures must continue to degrade gracefully.
- The visual design should not depend on ideal provider behavior.
- Frontend UI must continue to reflect explicit backend state rather than hidden assumptions.

### 9.4 Documentation continuity

- The version PRD should include a post-implementation section where future change notes can be appended.
- The folder structure should be simple enough to repeat for `v0.4`, `v0.5`, and beyond.

## 10. Technical Guardrails

### 10.1 Ownership boundaries

- `app.py` coordinates routes, session state, and API serialization.
- `game_logic.py`, `game_logic_restriction.py`, and `game_logic_blocks.py` own gameplay rules.
- `llm_client.py` owns provider prompts, parsing, validation, and fallback behavior.
- `frontend/src/*` owns browser rendering and orchestration.
- `templates/*.html` stay as shells, not application logic dumps.

### 10.2 Mode boundaries

- Do not collapse the three modes into one oversized resolver or one oversized controller.
- Shared helpers should only be extracted when the abstraction is genuinely cleaner than duplication.

### 10.3 Contract discipline

- Do not quietly duplicate backend gameplay rules in the frontend.
- Keep state payloads explicit and mode-aware.
- Avoid catch-all utility files that turn into dumping grounds.

### 10.4 Documentation discipline

- `docs/PRD.md` remains the stable foundation.
- `PRDs/v0.3/` is the version-specific iteration packet.
- release notes and actual implementation history still belong in `CHANGELOG.md` and `docs/V*.md`

## 12. Acceptance Criteria

This iteration packet is acceptable if:

- the repo has a repeatable versioned PRD structure
- the `v0.3` PRD explains product direction, scope, and guardrails clearly
- the demo HTML provides a credible visual target for future implementation
- the design direction matches the product's calm, sharp, playful identity
- the iteration materials are understandable without rereading the entire repo history
