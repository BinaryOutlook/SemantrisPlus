## 1. Iteration Summary

`v0.5` is the HMAS readiness and architecture modernization iteration for
Semantris Plus.

The project is already playable, tested, and stronger than a prototype. The next
problem is command structure. Two files now carry too much operational load:

- `app.py` owns routes, session state, serialization, persistence metadata,
  vocabulary selection, and mode turn orchestration.
- `llm_client.py` owns prompts, provider SDK wrappers, parsing, fallback
  strategy, cache policy, diagnostics, fake ranking, and startup probing.

The goal of `v0.5` is to make the codebase easier for exactly five parallel
Platoon Commander agents to extend without stepping on each other.

This is not a framework rewrite. This is a controlled refactor and execution
readiness pass that preserves the current Flask, Jinja, TypeScript, esbuild,
SQLAlchemy, and provider architecture while separating responsibilities into
clear command zones.

## 2. Current Baseline

Before `v0.5`, the repo has:

- three playable modes: Iteration, Restriction, and Blocks
- mode-specific backend rule modules
- Flask-rendered page shells plus JSON APIs
- TypeScript browser clients with esbuild bundles
- light, dark, and Cupertino themes
- Gemini and OpenAI-compatible ranking support
- deterministic fake-ranker mode for tests
- semantic cache abstraction with in-memory support
- local semantic fallback and older heuristic fallback
- SQLAlchemy-backed local run persistence
- backend unit tests, frontend unit tests, Biome linting, and Playwright E2E
  flows

The validation baseline observed during `v0.5` planning:

| Command | Result |
| --- | --- |
| `./.venv/bin/python -m unittest discover -s tests` | Passed, 65 tests |
| `npm run check:frontend` | Passed |
| `npm run test:frontend` | Passed, 14 tests |
| `npm run lint` | Passed |
| `npm run test:e2e` | Passed, 4 tests |

Important local-tooling note:

- `python3 -m unittest discover -s tests` can fail outside the project virtualenv
  because system Python may not have SQLAlchemy installed.
- Backend validation should use `./.venv/bin/python -m unittest discover -s tests`
  unless the environment has been explicitly rebuilt.

## 3. v0.5 Thesis

The next product gains will come from better command boundaries, not from a
large replatform.

`v0.5` should:

1. reduce congestion in `app.py`
2. split semantic ranking into a maintainable module group
3. introduce clearer typed contracts at system boundaries
4. consolidate duplicated frontend infrastructure
5. add enough CI, docs, and verification discipline for a five-agent HMAS
   workflow

The guiding rule is:

keep the game moving after every checkpoint.

Each phase must leave the app playable and the test suite green before the next
Platoon Commander receives broader authority.

## 4. Goals

### 4.1 Product goals

- keep all three current modes playable
- preserve the existing player-facing behavior unless explicitly called out
- improve reliability signals around ranking, fallback, and local bests
- make future game-feel and leaderboard work safer to add
- create a command board that lets future agents understand ownership quickly

### 4.2 Engineering goals

- delegate `app.py` responsibilities into smaller modules without changing API
  behavior
- split `llm_client.py` into semantic-engine responsibilities while preserving
  a compatibility import surface during migration
- reduce duplicated frontend API and HUD infrastructure
- keep tests deterministic and independent from live remote model access
- add or update documentation in the same pass as meaningful structural changes

### 4.3 HMAS goals

- define exactly five Platoon Commander lanes
- give each platoon an explicit write boundary
- identify specialized Shadow Striker subtasks before implementation starts
- prevent duplicate ownership of high-risk files
- make checkpoints and validation commands non-negotiable

## 5. Non-Goals

`v0.5` is not trying to deliver:

- a React, Vue, Svelte, Next.js, FastAPI, or Django migration
- a separate SPA
- cloud accounts, multiplayer, or live-service infrastructure
- a generic agent orchestration platform inside the game
- major gameplay rule changes to Iteration, Restriction, or Blocks
- removal of Gemini or OpenAI-compatible support
- removal of existing light, dark, or Cupertino themes
- a dependency-heavy semantic stack unless it is justified by a narrow fallback
  or evaluation task

### 6.5 Contested-file protocol

`app.py` and `llm_client.py` are restricted command posts until their
responsibilities are extracted.

- Only one Platoon Commander may hold active write authority for `app.py` at a
  time.
- Only one Platoon Commander may hold active write authority for `llm_client.py`
  at a time.
- Shared test files may be edited by the platoon that owns the behavior under
  test, but the handoff must say which contract was changed.
- README, PRD, and workflow updates close the phase; they should not get ahead
  of unmerged code behavior.
- Every handoff report must include files changed, commands run, contract
  changes, and known follow-up risk.

### 6.6 Rules of engagement for agents

- Read `docs/PRD.md`, this file, and `PRDs/v0.5/v0.5-demo.html` before coding.
- Read the relevant current code before proposing edits.
- Keep changes within assigned platoon ownership unless the Theater Commander
  approves a boundary crossing.
- Do not refactor and add unrelated product behavior in the same patch.
- Preserve public API payloads unless this PRD explicitly authorizes an additive
  field.
- Update tests with every meaningful behavior or contract change.
- Update docs when file ownership, commands, or architecture boundaries change.
- After each checkpoint, run the required validation commands before advancing.

## 7. Target Architecture

### 7.1 Preserve the stack

The target remains:

- Flask app
- Jinja templates
- JSON APIs
- vanilla TypeScript modules
- esbuild bundles
- custom CSS
- Pydantic Settings
- SQLAlchemy run persistence
- Gemini and OpenAI-compatible semantic providers

### 7.2 Backend route layer target

`app.py` should remain a recognizable entry point, but it should delegate most
non-route responsibilities.

Recommended extraction targets:

| Responsibility | Current source | Recommended target |
| --- | --- | --- |
| Vocabulary catalog loading | `app.py` | `vocabulary.py` |
| Mode ids and page routing metadata | `app.py` | `mode_registry.py` |
| Session selection and state commits | `app.py` | `session_state.py` |
| API state serialization | `app.py` | `state_serializers.py` |
| Persistence metadata attachment | `app.py` | `run_lifecycle.py` or `state_serializers.py` |
| Iteration and Restriction turn orchestration | `app.py` | `turn_services.py` |
| Blocks primary selection and frontier scoring | `app.py` | `blocks_orchestration.py` |
| Flask route functions | `app.py` | keep in `app.py` first, then consider blueprints |

Blueprints are optional for `v0.5`. The first victory condition is a thinner
entry point with stable route behavior, not a directory reshuffle for its own
sake.

### 7.3 Semantic engine target

`llm_client.py` should become a compatibility facade or a much smaller module.

Recommended module group:

```text
semantic_engine/
├── __init__.py
├── contracts.py
├── prompts.py
├── parsers.py
├── diagnostics.py
├── fallback.py
├── resilient.py
├── factory.py
└── providers/
    ├── __init__.py
    ├── gemini.py
    └── openai_compatible.py
```

Migration rule:

- Keep existing imports from `llm_client.py` working until all tests and app
  imports have been updated deliberately.
- Do not move provider behavior and change ranking semantics in the same patch.
- Parser tests should move with parser code, not be deleted or weakened.

### 7.4 Backend state contracts target

Current backend state is mostly dictionary-based. `v0.5` should improve contract
clarity without overbuilding.

Acceptable approaches:

- `TypedDict` for serialized state payloads
- dataclasses for pure gameplay state where mutation is currently hard to trace
- Pydantic models for API payload validation only where startup or request-bound
  validation adds real safety

Guardrail:

- Do not force Iteration, Restriction, and Blocks into one generic state model.
  Shared state fields may be typed once, but mode-specific fields stay explicit.

### 7.5 Frontend target

The frontend should remain vanilla TypeScript, but shared infrastructure should
stop being copied across modes.

Recommended extraction targets:

| Responsibility | Current source | Recommended target |
| --- | --- | --- |
| Repeated `fetchJson` helpers | `api.ts`, `restriction_api.ts`, `blocks_api.ts` | `api_client.ts` |
| Tower run lifecycle | `controller.ts`, `restriction_controller.ts` | shared tower lifecycle helpers |
| Timer and game-over helpers | `hud.ts`, `blocks_controller.ts` | shared timer/modal utilities |
| Persistence message formatting | `hud.ts`, `blocks_controller.ts` | shared persistence UI helper |
| Repeated template head/topbar/modal | templates | `templates/partials/` |

Guardrail:

- Do not make a generic controller so abstract that mode behavior becomes harder
  to read than the current duplication.

## 8. In-Scope Workstreams

### 8.1 Workstream A: App command-post decomposition

Purpose:

- make `app.py` small enough that route behavior is obvious
- let future agents work on vocabulary, session, serialization, and turns
  without editing one giant file

Required outcomes:

- extracted vocabulary catalog module
- extracted session/state lifecycle module
- extracted API serialization module
- extracted Blocks orchestration module
- unchanged public endpoints
- unchanged current Playwright flows

### 8.2 Workstream B: Semantic engine package

Purpose:

- split provider, parsing, fallback, cache, and diagnostics responsibilities
- make ranking behavior easier to test and tune
- make future semantic upgrades possible without another giant-file expansion

Required outcomes:

- semantic contracts are explicit
- parser functions live apart from provider SDK wrappers
- provider wrappers live apart from resilient fallback policy
- `FakeRanker`, semantic fallback, and heuristic fallback remain testable
- cache behavior remains validated and provider-safe

### 8.3 Workstream C: Typed contracts and state safety

Purpose:

- reduce accidental state drift between gameplay, Flask session, API payloads,
  and frontend types

Required outcomes:

- documented backend state contract for each mode
- additive or non-breaking API payload changes only
- frontend types stay aligned with serialized backend payloads
- test coverage catches missing required fields for each mode

### 8.4 Workstream D: Frontend consolidation and UX readiness

Purpose:

- reduce duplicated frontend infrastructure
- make future UI/game-feel work faster and safer

Required outcomes:

- shared API client
- shared persistence message formatting
- shared timer or modal utilities where clear
- Jinja partials for repeated head/theme/topbar/game-over markup if they reduce
  duplication without hiding page identity
- no visual regressions in current flows

### 8.5 Workstream E: QA, CI, dependency hygiene, and docs

Purpose:

- make the project runnable by future humans and agents without tribal memory
- automate the validation baseline that already exists

Required outcomes:

- documented validation command set
- CI workflow for Python tests, frontend typecheck, Vitest, Biome, build, and
  Playwright where practical
- dependency audit notes for runtime versus non-runtime packages
- README and PRD updates after structural changes

- Existing routes must keep working:
  - `/`
  - `/iteration-mode`
  - `/restriction-mode`
  - `/blocks-mode`
  - `/api/game/state`
  - `/api/game/new`
  - `/api/game/turn`

  - `/api/restriction/new`
  - `/api/restriction/turn`
  - `/api/blocks/state`
  - `/api/blocks/new`
  - `/api/blocks/turn`
- Current API payload fields remain available.
- Any new API fields must be additive and documented in TypeScript types.
- Current local fake-ranker E2E path must remain deterministic.

### 9.2 Gameplay requirements

- Iteration Mode behavior must remain unchanged unless a task explicitly says
  otherwise.
- Restriction Mode behavior must remain unchanged unless a task explicitly says
  otherwise.
- Blocks Mode behavior must remain unchanged unless a task explicitly says
  otherwise.
- Game-over persistence behavior must keep saving completed runs when enabled.
- Disabled persistence must not block gameplay.

### 9.3 Ranking requirements

- Remote provider output must still be validated before use.
- Cache hits must not bypass structural validation discipline.
- Fallback warnings must remain visible to the frontend.
- The startup probe must remain optional and skip-safe for local tests.
- Provider-specific configuration errors must continue producing clear
  diagnostics.

### 9.4 Frontend requirements

- Theme selection must continue to persist across pages.
- All three game modes must still support keyboard clue submission.
- Reduced-motion behavior must remain respected.
- Game-over modal behavior must remain reachable.
- Build output must continue landing in `static/js/` unless the PRD is amended.

## 13. Risks And Mitigations

| --- | --- | --- |
| Refactor changes gameplay behavior accidentally | High | Add state contract tests and run E2E after each backend checkpoint |
